# Purpose

This document defines the scope of the Spot & Report Build Week Minimum Viable Product (MVP).

Its purpose is to establish a clear, shared understanding of what will be delivered during Build Week while preventing unnecessary scope expansion.

The MVP is intentionally focused.

Every included feature should contribute directly to demonstrating the core value of the platform.

Ideas that fall outside this document are not rejected—they are intentionally deferred until the MVP has been successfully completed.

---

# Relationship to the Vision

The long-term vision for Spot & Report is to become a flexible platform for reporting real-world incidents.

The Build Week MVP represents the first practical implementation of that vision.

Rather than attempting to solve every reporting scenario, the MVP demonstrates one complete reporting workflow that proves the underlying platform concept.

Every implementation decision should support the long-term vision without expanding the short-term scope.

---

# Build Week Objective

Deliver a polished, production-quality demonstration that proves public incident reporting can be dramatically simplified through thoughtful user experience and responsible artificial intelligence.

The MVP should showcase one complete reporting journey that feels intuitive, trustworthy and ready for real-world adoption.

Success is measured by quality, usability and completeness—not by feature count.

---

# Definition of MVP

For Spot & Report, a Minimum Viable Product is not the smallest amount of software that can be built.

It is the smallest complete experience that convincingly demonstrates the value of the product.

The MVP should feel intentional, reliable and ready to demonstrate.

It is always preferable to deliver one exceptional workflow than several incomplete ones.

---

# Primary Success Criteria

The Build Week MVP will be considered successful if it demonstrates that:

- A first-time user can successfully report a sick or dead bird.
- The complete workflow can be completed in approximately sixty seconds.
- AI meaningfully improves report quality without increasing complexity.
- The experience feels intuitive, trustworthy and polished.
- The platform architecture clearly supports future expansion.

---

# Target Users

The Build Week MVP is designed for members of the public who encounter a sick or dead wild bird.

Users should not require:

- Technical knowledge.
- Knowledge of government reporting systems.
- Previous experience using the application.

The experience should feel approachable regardless of age or technical confidence.

---

# Core User Journey

The MVP supports the following reporting journey:

1. Open Spot & Report.
2. Select the Bird Report workflow.
3. Capture or upload a photo.
4. Confirm the incident location.
5. Answer a small number of guided questions.
6. Review the AI-assisted report.
7. Submit the completed report.
8. Receive confirmation that the report has been successfully prepared and submitted.

Every stage should feel clear, calm and require minimal effort.

# Included AI Capabilities

Artificial intelligence is included to improve report quality—not to make reporting decisions.

The MVP includes AI assistance for:

- Image interpretation.
- Report drafting.
- Information structuring.
- Writing clarity.
- Consistency improvements.

AI should never invent information that cannot reasonably be supported by user-provided evidence.

Users remain responsible for reviewing reports before submission.

# Design Constraints

Design Constraints define the characteristics that every future feature must preserve.

They are intentionally difficult to change because they protect the simplicity, consistency and usability of the platform.

New functionality should adapt to these constraints rather than requiring the constraints to change.

The Build Week MVP should maintain the following characteristics:

## Simplicity

- Reporting should be achievable in approximately sixty seconds.
- Every screen should contribute directly to completing a report.
- Remove unnecessary decisions wherever possible.

## User Experience

- Mobile-first design.
- Minimal typing.
- Small number of guided questions.
- Clear progress indication.
- Calm, government-neutral visual design.

## Artificial Intelligence

- AI should quietly assist rather than dominate the experience.
- AI should explain, not surprise.
- Users always remain in control of submitted information.

## Product Scope

- One complete workflow is preferred over multiple incomplete workflows.
- New features must reduce friction rather than increase capability for its own sake.

# MVP Exit Criteria

The Build Week MVP is considered complete only when all of the following statements are true.

## Product

- The complete Bird Report workflow can be demonstrated from beginning to end.
- The application successfully demonstrates the core value of Spot & Report.
- A first-time user can complete the workflow without explanation.

## Quality

- The implementation satisfies the project's Definition of Done.
- The application is stable enough for public demonstration.
- No known issue significantly impacts the primary user journey.

## Documentation

- Project documentation accurately reflects the implementation.
- Outstanding work is captured as future opportunities rather than unfinished MVP functionality.

## Demonstration

- The application is ready for the Build Week demonstration.
- The complete workflow can be presented confidently without relying on unfinished functionality.

The MVP is complete when the product demonstrates a polished, reliable and convincing end-to-end experience.

It is not complete simply because development time has ended.

# Final Principle

Every feature added to the Build Week MVP should answer one question:

**Does this make it easier for someone to successfully report a real-world incident?**

If the answer is no, it does not belong in the MVP.

Protect the scope.

Deliver one exceptional experience.

Earn the opportunity to expand.
